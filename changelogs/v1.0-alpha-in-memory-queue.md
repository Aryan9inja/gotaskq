# GoTaskQ v1.0-alpha-in-memory-queue

Release date: 2026-04-17
Release type: Alpha

## Summary
This release introduces the first complete in-memory runtime path for GoTaskQ. A job can be accepted over HTTP, persisted in an in-memory store, scheduled in an in-memory priority queue, and executed asynchronously by a worker pool. The release focuses on runtime correctness, clear interface boundaries, and backend swap readiness.

## Architecture and Execution Flow
1. API ingress
   - `POST /jobs` decodes request payload, validates required fields, allocates a Snowflake ID, and assigns initial state (`PENDING`).
   - Job records are written to the store before enqueue to preserve lifecycle consistency for fetch operations.
2. Scheduling and queueing
   - Jobs are inserted into a heap-backed queue.
   - Effective execution time is managed through `run_after`; delayed jobs are not dequeued until ready.
3. Worker execution
   - A configurable worker pool pulls ready jobs from the queue.
   - Worker transitions status to `RUNNING`, executes the registered handler, and finalizes with `DONE` or `FAILED`.
4. Retrieval
   - `GET /jobs/{id}` reads the latest in-memory job snapshot by ID.

## Technical Additions
- End-to-end in-memory lifecycle: submit -> persist -> schedule -> execute -> fetch status.
- Deterministic scheduler ordering:
  - earliest `run_after`
  - higher `priority` for equal `run_after`
  - earlier `created_at`
  - lexicographic ID tie-break for deterministic dequeue behavior
- Concurrency model:
  - worker pool uses goroutines + context cancellation for lifecycle control
  - queue/store use mutexes for concurrent access safety
- Interface seams for backend evolution:
  - queue abstraction (`Enqueue`, `Dequeue`, `Len`, `Name`)
  - store abstraction (`Save`, `Get`, `UpdateStatus`, `Delete`)
  - handler lookup abstraction for job-type dispatch
- Snowflake IDs integrated as monotonic identifiers suitable for distributed patterns.
- Retry foundation implemented:
  - retry eligibility by `retry_count < max_retries`
  - exponential backoff (`delay * 2^retry_count`)
  - capped by `MAX_DELAY`
  - random jitter to reduce synchronized retries

## API
- `POST /jobs`
  - Request fields: `type`, `payload`, `priority`, `delay`, `max_retries`.
  - Validation: non-empty `type`, `priority` in range `0..10`.
  - Behavior: generates ID, stores job, enqueues job to default queue, returns created job object.
- `GET /jobs/{id}`
  - Returns job state from in-memory store for the provided ID.

## State and Scheduling Semantics
- Job status model: `PENDING`, `RUNNING`, `DONE`, `FAILED`, `DEAD`.
- Current transition constraints are enforced in store status updates.
- Delayed execution is implemented with `run_after`; queue returns "not ready" when the head job is scheduled in the future.

## Configuration
- `PORT` (default: `8000`)
- `MAX_RETRIES` (default: `5`)
- `BASE_DELAY` (default: `100`)
- `MAX_DELAY` (default: `5000`)
- `NUM_WORKERS` (default: `10`)

## Operational Characteristics
- Persistence model: process-local memory for queue and store.
- Startup behavior: creates default queue, registers handlers, starts worker pool, starts HTTP server.
- Shutdown behavior: worker pool receives cancellation and waits for goroutines to drain.

## Known Limitations
- In-memory only persistence; state is lost on restart.
- Retry engine exists but is not fully integrated in worker failure-path execution.
- DLQ path is not fully implemented.
- Metrics are scaffolded but not instrumented.
- Queue context cancellation is not yet enforced in enqueue/dequeue operations.
- API surface is intentionally minimal for alpha (`POST /jobs`, `GET /jobs/{id}`).
