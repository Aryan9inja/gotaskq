# GoTaskQ v1.0-beta-redis-based-queue

Release date: 2026-04-27
Release type: Beta

## Summary
This release moves GoTaskQ from an in-memory-first alpha path to a Redis-backed beta runtime. Jobs can now be persisted in Redis, scheduled via a Redis priority queue, executed by workers with retry handling wired into failure paths, and recovered through DLQ operations.

## What Changed Since v1.0-alpha-in-memory-queue
1. Redis backend support is now available at runtime.
   - Server startup selects backend using `USE_REDIS` and `REDIS_URL`.
   - Redis connectivity is validated during startup with `PING` and fails fast on misconfiguration.
2. Redis job persistence was added.
   - Job records are stored as Redis hashes (`gotaskq:job:{id}`) with lifecycle fields.
   - Status transitions are validated atomically via Lua in Redis (`PENDING -> RUNNING -> DONE/FAILED`, `FAILED -> PENDING/DEAD`).
3. Redis queue implementation was added.
   - Pending jobs are tracked in Redis Sorted Set by `run_after` score.
   - Payloads are stored in Redis Hash and removed atomically on dequeue.
   - Dequeue uses a Lua script for ready-check + pop semantics.
4. Worker notification flow improved for Redis queues.
   - Redis queue publishes enqueue notifications.
   - Worker pool subscribes and wakes on queue notifications instead of only polling sleeps.
5. Retry flow is now integrated into worker failure handling.
   - Handler errors, missing handler cases, and panic recovery paths invoke retry handling.
   - Retry engine updates state to `PENDING` and re-enqueues eligible jobs.
   - Retry exhaustion transitions jobs to `DEAD` and writes them to DLQ.
6. DLQ implementation and APIs were completed for Redis-backed runs.
   - Added DLQ storage operations: save, get, list, delete.
   - Added HTTP endpoints:
     - `GET /dlq`
     - `POST /dlq/{id}/replay`
     - `DELETE /dlq/{id}`
   - Replay resets dead jobs to `PENDING`, re-enqueues them, then removes them from DLQ.
7. Backend seams from alpha are now exercised in production path wiring.
   - Queue, store, retry, and DLQ interfaces are now used across both memory and Redis modes.

## Architecture and Execution Flow (Beta)
1. API ingress
   - `POST /jobs` validates input, allocates Snowflake ID, and saves job state.
2. Persistence
   - Job state is persisted in Redis job store when Redis mode is enabled.
3. Scheduling and queueing
   - Jobs are inserted into Redis sorted set keyed by `run_after` and deterministic member ordering.
4. Worker execution
   - Worker updates status to `RUNNING`, executes handler, and finalizes `DONE` on success.
   - On failure paths, retry engine decides re-enqueue or dead-letter outcome.
5. Dead-letter recovery
   - Exhausted jobs are marked `DEAD` and persisted in Redis DLQ.
   - Operators can list, replay, or delete dead jobs through DLQ APIs.

## API
- `POST /jobs`
  - Request fields: `type`, `payload`, `priority`, `delay`, `max_retries`.
  - Validation: non-empty `type`, `priority` in range `0..10`.
  - Behavior: creates job, stores it, and enqueues into default queue.
- `GET /jobs/{id}`
  - Returns the current job snapshot by ID.
- `GET /dlq`
  - Lists dead jobs from DLQ.
- `POST /dlq/{id}/replay`
  - Replays a dead job back into the default queue.
- `DELETE /dlq/{id}`
  - Removes a dead job from DLQ.

## Configuration
- Existing:
  - `PORT` (default: `8000`)
  - `MAX_RETRIES` (default: `5`)
  - `BASE_DELAY` (default: `100`)
  - `MAX_DELAY` (default: `5000`)
  - `NUM_WORKERS` (default: `10`)
- Added for beta backend selection:
  - `USE_REDIS` (`true` enables Redis backend)
  - `REDIS_URL` (required when `USE_REDIS=true`)

## Operational Characteristics
- Redis mode:
  - Durable job/queue/DLQ state in Redis (subject to Redis deployment durability settings).
  - Worker wake-up on queue notifications.
- Memory mode:
  - In-memory queue/store behavior remains available for local development.

## Known Limitations
- Metrics are still scaffolded and not instrumented.
- Single default queue wiring remains the active runtime path.
- DLQ endpoints require a configured DLQ store; in-memory mode does not currently provide DLQ storage.
- Retry backoff cap is enforced, but `BASE_DELAY` is not yet wired as a global retry baseline.
